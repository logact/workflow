## 功能分解的workflow

### 角色
1. 需求分析师 用于对齐用户的需求和agent，用户与其对抗不断输出向用户确认是否agent得到是用户想要的。
2. 架构师，负责输出大概如何实现，不输出具体的代码实现，简要的输出如何实现的这个功能，与用户对抗对齐，接收用户的质询。
4. 项目经理，项目经理接收架构师的实现指导，拆分具体的任务（任务包括，任务ID，任务验收标准，依赖的任务，任务实现者，任务标签，交付产物）。同样的需要接受用户的质询和用户的确认对齐。
   1. 这里生成的任务可能的标签有后端任务，前端任务，设计任务
5. 测试开发 在架构师输出实现方案后介入，根据需求文档和实现方案生成覆盖全部功能点的测试任务代码（含 E2E 测试文件），提交用户评审
6. 后端开发：负责实现后端代码
7. 设计师：设计UI 
8. 前端任务：负责前端任务
9. 后端检验者：假定后端实现的是错误的，检验后端的代码根据设定的准则，输出检验错误
10. 设计检验者：假定设计师实现的是错误的，检验后端的代码根据设定的准则，输出检验错误
11. 前端检验者：假定前端实现的是错误的，检验前端的代码根据设定的准则，输出检验错误
12. 应用者：更具检验者的建议修改

### workflow 过程

#### phase1 规划阶段
1. 用户提出需求
2. 需求分析师分析用户的需求明确范围，规范化需求与用户对齐，输出需求文档
3. 架构师根据需求分析师的需求设计实现方案
4. **测试开发在架构师输出实现方案后介入**：根据需求文档和架构方案生成覆盖全部功能点的测试任务代码（E2E 测试），交给用户评审确认
5. 项目经理在测试任务通过评审后，将测试任务纳入任务图，拆分后端/前端/设计实现任务并设置测试任务对实现任务的依赖

phase1 规划阶段最终输出带依赖项的任务图（包含预生成的测试任务）
#### 实现阶段
开始启动创建新的分支

多个 kimi agent 实例启动就像在 bun 迁移到 rust 的 blog 中写的那样，我们根据任务的标签分别映射调用不同的 loop（后端 loop、前端 loop、设计 loop、测试 loop）

#### 本地验证
每次跑完一个任务都要在本地测试通过然后提交

### CI验证
这个需求的所有任务跑完直接提交PR 一直到PR成功
