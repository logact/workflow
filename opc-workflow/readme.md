我从bun的迁移过程收益良多https://bun.com/blog/bun-in-rust#compiler-errors-as-a-work-queue，多agent使用任务队列 来统一协调资源消费，通过隔离的shard资源来提高并发。我也想要创建这样的workflow


## 功能分解的workflow

### 角色
1. 需求分析师 用于对齐用户的需求和agent，用户与其对抗不断输出向用户确认是否agent得到是用户想要的。
2. 架构师，负责输出大概如何实现，不输出具体的代码实现，简要的输出如何实现的这个功能，与用户对抗对齐，接收用户的质询。
4. 项目经理，项目经理接收架构师的实现指导，拆分具体的任务（任务包括，任务ID，任务验收标准，依赖的任务，任务实现者，任务标签，交付产物）。同样的需要接受用户的质询和用户的确认对齐。
   1. 这里生成的任务可能的标签有后端任务，前端任务，设计任务
5. 测试开发 测试开发根据项目需求文档啊以及实现方案设计测试用例
6. 后端开发：负责实现后端代码
7. 设计师：设计UI 
8. 前端任务：负责前端任务
9. 后端检验者：假定后端实现的是错误的，检验后端的代码根据设定的准则，输出检验错误
10. 设计检验者：假定设计师实现的是错误的，检验后端的代码根据设定的准则，输出检验错误
11. 前端检验者：假定前端实现的是错误的，检验前端的代码根据设定的准则，输出检验错误
12. 应用者：更具检验者的建议修改

### workflow 过程

#### phase1 规划阶段
1. 用户提出需求
2. 需求分析师分析用户的需求明确范围，规范化需求与用户对齐，输出需求文档
3. 架构师更具需求分析师的需求设计实现方案
4. 项目经理根据设计方案和设计文档拆分任务输出待依赖项的任务

phase1 规划阶段最终输出待依赖项目的任务图
#### 实现阶段
开始启动创建新的分支

测试开发介入创建测试用例（e2e测试功能）

多个kimi agent 实例启动就像在bun 迁移到rust 的blog 中写的那样,我们根据任务的标签分别映射调用不同的loop（后端loop,前端loop,设计loop） 

#### 本地验证
每次跑完一个任务都要在本地测试通过然后提交

### CI验证
这个需求的所有任务跑完直接提交PR 一直到PR成功
